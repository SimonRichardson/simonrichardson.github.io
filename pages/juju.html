juju

Juju is an open source application modeling tool. Juju focuses on reducing the operation overhead of today's software by facilitating quickly deploying, configuring, scaling, integrating, and performing operational tasks on a wide choice of public and private cloud services along with bare metal servers and local container based deployments.

Recent work (2023–2026)

Key areas:

  • Architecture and layering — Led the introduction of a clean layered architecture across the codebase: domain types, services, and state (persistence) as distinct concerns. This gave the project testable boundaries, clearer ownership of business logic, and a path away from the monolithic facade pattern that had accumulated over years of development. [1][2]
  • Database migration from MongoDB to Dqlite — Helped architect and implement the transition of Juju's persistence layer from MongoDB to Dqlite (a distributed SQLite). This touched nearly every subsystem — designing schemas, writing DDL migrations, building the database worker infrastructure, and ensuring upgrade re-entrancy so controllers could move forward safely. [1][2]
  • Object store — Built the object store subsystem from the ground up, providing blob storage for charms and resources. Implemented both local file-backed and S3-compatible backends, wired through the API server and machine manifolds, with a focus on correctness around hashing and content verification. [1][2][3]
  • Tracing and observability — Introduced OpenTelemetry tracing throughout the system. Built a dedicated tracer worker, threaded trace and span context through the API client layer, and enabled charm-level tracing configuration via the control socket. This gave operators visibility into request flows across controllers, agents, and workloads for the first time. [1][2][3]
  • Technical leadership — Guided architectural decisions across the team, reviewed and shaped the direction of major subsystems, and mentored engineers through complex refactoring work. Drove conventions around testing patterns, error handling, and package structure that improved consistency across the project.
  • Worker and watcher infrastructure — Rebuilt core worker lifecycle management, including a new watcher registry, upgrade steps workers (controller and machine), the undertaker worker for model deletion, and the charm revision updater. Each designed with clean shutdown semantics and dependency injection. [1][2][3]
  • Model migration — Worked on model import/export and the migration target, including correct handling of model deletion, re-entrancy during upgrades, and ensuring the migration path worked end-to-end with the new database layer. [1][2][3]
  • API server — Reworked the API handler layer, removed legacy resource tracking, added authentication and authorisation to control socket HTTP endpoints, and improved connection handling for cross-model relations. Introduced a remote caller pattern for federated API communication. [1][2]
  • Provisioning and machine lifecycle — Optimised provisioning info retrieval, added machine placement and architecture handling, and introduced machine presence tracking via the status service. [1][2][3]
  • Upgrades — Designed the upgrade path for controllers and machines moving between major versions, including automatic model upgrades post-migration and ensuring the upgrade steps worker ran correctly on all agent types. [1][2][3]
